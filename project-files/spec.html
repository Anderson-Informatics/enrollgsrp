Wayne County PreK Enrollment System — Specification

Product Specification Document

Context: Wayne RESA and Subrecipients need a centralized system to manage PreK enrollment, ensuring eligible children receive high-quality preschool programming while complying with Michigan GSRP guidelines and FERPA requirements.

Overview

A web-based enrollment management system for Wayne County PreK programs that streamlines the application process from intake through enrollment. The system serves five user types: County Admins, District Admins, School Admins, Teachers, and Parents. It manages eligibility determination, document verification, waitlist prioritization, communications, and site visit scheduling. The system is not a student information system (SIS) but integrates with existing SIS platforms for data synchronization. It implements granular permission controls to ensure appropriate data access based on organizational hierarchy and compliance requirements.

Goals

  • Streamline the PreK enrollment process from initial application to final enrollment, reducing administrative burden on school staff and improving the experience for families
  • Automate eligibility determination based on GSRP guidelines (age, income, IEP status, housing status) with accurate priority tier assignment and timing restriction enforcement
  • Enable efficient document management with upload, verification, and correction request workflows that reduce paperwork and improve processing times
  • Provide real-time visibility into application status for parents while maintaining appropriate access controls for administrators
  • Support data integration with existing Student Information Systems (PowerSchool, MIStar, Infinite Campus) to eliminate manual data entry
  • Ensure FERPA compliance through audit logging, data encryption, and granular permission management across all user roles

User Roles

County Admin

Can do: Manage system-wide configuration, create and manage districts and district admins, upload and manage county-level PDF templates, view system-wide reports and analytics, add or remove custom fields to the county intake form (core eligibility fields cannot be removed), configure SSO settings for districts

Cannot do: Modify permissions of other County Admins or Super Admins, access school-level data without appropriate permissions

Authentication: Local authentication with option to configure SSO

District Admin

Can do: Full access to all school sites and students within their district, set up school sites and assign school admins, add teacher users to schools, enable multi-school enrollment support for teachers, create and maintain district-level enrollment forms using form builder, configure SIS integrations, upload district-specific PDF templates, monitor enrollment and waitlist status across district, generate and export reports for state submissions, add or remove custom priority questions to county intake form for their district schools, verify responses to district custom priority questions

Cannot do: Modify permissions of District Admins or higher-level users, access data from other districts

Authentication: District-specific SAML/SSO integration

School Admin

Can do: Manage school information, add/edit/remove teachers at their school, view and manage all applicant information received, coordinate with District Admin for school-level operations, oversee waitlist management for their school, monitor site visit scheduling and completion, generate school-specific reports, verify IEP documentation for applications indicating IEP status, verify district custom priority question responses for applications to their school

Cannot do: Modify permissions of School Admins or higher-level users, access data from other schools

Authentication: District-specific SAML/SSO integration

Teacher

Can do: Process and review applications for assigned school(s), if multi-school enrollment support permission is enabled access applications from multiple schools as configured, review enrollment form for completeness and verify all uploaded documents, mark each document as Verified, Needs Correction, or Missing, generate completed PDF forms for applicants, verify documents and complete enrollment verification during Stage 2, update enrollment status and communicate with parents, manage waitlist positions and openings, configure calendar availability for site visits after student enrollment, schedule and conduct site visits after student has started attending school, complete Ages and Stages Questionnaire (ASQ) assessment during site visits, verify IEP documentation and district custom priority question responses

Cannot do: Modify permissions of any other users, access applications from schools they are not assigned to

Authentication: District-specific SAML/SSO integration

Parent (Guardian)

Can do: Apply to PreK programs at any school site in the system, rank up to three school site preferences, apply for multiple children with pre-filled family data, complete pre-built county intake form with core required fields, provide child's date of birth for age eligibility determination, provide income information for income-based eligibility priority or decline to submit (may trigger enrollment hold until August 1), indicate IEP status and provide documentation if applicable, indicate housing status for priority determination, complete district-specific priority questions that display conditionally based on selected schools, complete any additional custom fields as configured by County Admins, upload required documents, save and resume applications before enrollment paperwork is submitted, complete district enrollment form immediately after intake submission, upload required documents and respond to document correction requests during enrollment paperwork stage, receive notifications for key events and status changes, view real-time application status including current stage and available actions, track document verification status and respond to correction requests, access communication history with teachers and school staff, schedule site visit after student has started attending school

Cannot do: Access other families' application data, modify system configuration, view applications for schools not selected in their preferences

Authentication: Hybrid authentication supporting both local username/password and OAuth (Google/Microsoft) options

Core Features

1. Granular Permission Management

Feature and function-specific permissions for all admin user types (County Admin, District Admin, School Admin, Teacher) with preset permission sets and custom configuration capabilities. Permissions are organized into logical groupings (System Management, District Management, School Management, Teacher, User Management, Data Access) with hierarchical control allowing higher-level users to manage permissions for lower-level users. Parent users have fixed permissions. The system enforces permission checks on all data access and feature usage to ensure FERPA compliance.

2. Authentication System

Flexible authentication supporting local accounts with username/password, OAuth integration (Google/Microsoft) for parents, and district-specific SAML/SSO for district staff. Password reset available for local accounts via email verification. District configuration includes setup instructions and required variable inputs (app IDs, credentials) for SAML/SSO integration.

3. Form Builder

Dynamic form creation tool with comprehensive field types (text, textarea, number, date, email, phone, address block, single select, radio group, checkbox, multi-select, file upload), conditional logic, validation rules, and multi-language support (English, Spanish, Arabic). Supports composite field groups, repeatable collection fields (household members, income sources), custom collection builder, field templates, form templates, role-based field access, and offline support with auto-sync. All form inputs are sanitized to prevent XSS attacks and SQL injection.

4. County Intake Form

Pre-built standardized form with core required fields (date of birth, income questions, IEP status, housing status, school selection/ranking) for automatic eligibility determination. County Admins can add or remove custom fields, but core eligibility fields cannot be removed. System automatically calculates eligibility, priority tier, and timing restrictions based on responses. District Admins can add custom priority questions that display conditionally when schools in their district are selected (e.g., sibling attendance verification). Responses to district custom priority questions must be verified by District or School Admins for priority status to apply.

5. Automatic Eligibility Determination

System automatically calculates age eligibility (must turn 4 on or before September 1st; children turning 4 between September 2 - December 1 can enroll after September 2nd if space permits), income-based priority (FPL 0-100% Tier 1, FPL 101-400% Tier 2, FPL 401%+ Tier 3 with enrollment hold until May 1st), and timing restrictions based on core required fields. Priority order: Housing status (homeless/foster care - highest) → Verified IEP → Verified district custom priorities → FPL tiers. If parent declines to submit income information, enrollment placed on hold until August 1st.

6. Application Lifecycle Management

Four-stage enrollment process: Stage 1 (Intake) - Parents submit county intake form, School Admin assigns teacher, application routes to school; Stage 2 (Enrollment Paperwork) - Parents complete district enrollment form and upload documents, Teacher verifies all materials; Stage 3 (Enrolled) - Teacher marks student as enrolled, assigns to classroom, student begins attending school; Stage 4 (Site Visit) - Teacher configures calendar, Parent schedules visit, Teacher conducts ASQ assessment. Each stage triggers automated notifications to appropriate parties. Parents have real-time visibility into current stage, available actions, waiting status, progress bar, and stage-specific details.

7. District Enrollment Form

District-specific enrollment forms created by District Admins using the form builder, completed by parents immediately after intake submission. Parents complete district-specific questions and upload required documents (proof of residency, income verification, immunization records, custody documents). Teachers review form completeness and verify all uploaded materials.

8. Document Upload & Verification

Document submission workflow where parents upload required documents and teachers verify each document with status tracking. System validates file types and converts images to PDF on mobile devices. Teachers review each document and mark as Verified, Needs Correction, or Missing, with ability to request specific corrections with explanations. Parents receive notifications and upload corrected documents. Real-time status display for parents (Verified, Pending, Action Required) with audit trail.

9. PDF Auto-Fill

Automatic population of PDF enrollment forms with applicant data, eliminating manual data entry and reducing errors. Admins upload fillable PDF templates and map form fields to system fields. Teachers generate completed PDFs with one click from applicant records. System automatically populates PDF with applicant data including calculated fields. Supports multiple templates, field mapping interface, smart field matching, custom field mapping, multi-value support for repeating fields, and batch processing.

10. Capacity & Waitlist Management

School capacity tracking with automated waitlists that fill based on priority when spots become available. District Admins configure capacity for each school in their district. System automatically routes applications to first-choice school with available capacity. Applications for full schools are placed on automated waitlists. When spots open, system fills from waitlist based on priority order. System maintains 10% allocation hold for 0-400% FPL students until September 1.

11. Teacher-Parent Communications

Multi-channel communication system enabling teachers to contact parents via email, SMS, and logged phone calls with centralized history tracking. Teachers send emails or SMS directly from applicant records using templates or custom messages. Parent replies are automatically logged in communication history. Teachers log phone calls and in-person meetings with notes and outcomes. Notification badge alerts teachers to unread replies. System automatically tracks date of last contact for reporting. Contact activity dashboard available for administrators.

12. Site Visit Scheduling

Teachers configure calendar availability with travel buffer times after student enrollment and during early school year. Parents self-schedule from teacher's available time slots after student has started attending school. Parents can reschedule up to 24 hours before visit if needed. Teachers confirm visit location (home, library, community center). Site visits must be completed in private space, never in public spaces. Initial visit lasts 45 minutes and includes Ages and Stages Questionnaire (ASQ) assessment, discussion of child's progress and development, sharing curriculum and program information, and setting goals for the child.

13. Ages and Stages Assessment

Teachers complete Ages and Stages Questionnaire (ASQ) assessment during site visits. Assessment covers child development milestones and helps identify areas where the child may need additional support. Second visit after assessment checkpoint reviews individual child needs and kindergarten transition.

14. Classroom Assignment

After student is fully enrolled with all documents verified, teachers assign students to specific classrooms based on age, IEP status, and program requirements. Ensure classroom ratios are maintained (typically 10:1 or 12:1). Parents receive official classroom assignment notification with teacher contact information. Orientation scheduling coordinated by assigned teacher.

15. Multi-language Support

Forms support multiple languages (English, Spanish, Arabic) with RTL layout for Arabic. Admins can configure language-specific content for form fields and templates. System detects browser language preference where possible.

16. Multi-child Applications

Parents can apply for multiple children with pre-filled family data from first application to reduce data entry. Separate applications per child with shared parent/guardian information automatically populated.

17. Application Transfers

County Admins can transfer any applicant to any school. District Admins can transfer applicants to another school in their district if selected by parent. Transfers preserve all application data, documents, and verification status. Students may be enrolled at one school while waitlisted for a higher-ranked school preference.

18. Automated Notifications

Email + SMS notifications to parents for key events (application received, placement, site visit, enrollment complete, document requests). Teachers receive notifications for new applications, document submissions, communication replies. Notification center provides centralized view of all alerts.

19. Reporting

Enrollment statistics, waitlist reports, demographics reporting, contact activity dashboard, last contact reports showing days since last contact for all applicants, communication volume metrics, response rate tracking. Reports available at system-wide, district-wide, and school-wide levels based on user permissions.

20. Data Integration

Phase 1: CSV exports for manual SIS import. Phase 2: Direct API integration with PowerSchool, MIStar (Wayne County custom SIS), and Infinite Campus. For 25-26 SY, Community Based Organizations will utilize MiSTAR for enrollment. Field mapping configuration for SIS integration with synchronization status monitoring and error handling.

21. Security & Audit Logging

FERPA-focused audit logging for data access events, document uploads and downloads, enrollment decisions and status changes, and admin configuration changes. All audit logs include timestamp, user, action, and affected records. Sensitive data (SSN, income information) encrypted at rest. CSRF protection on all form submissions. Secure file uploads with virus scanning, type validation, and secure storage. Continuous vulnerability monitoring, annual penetration testing, automated security scanning in CI/CD pipeline. Incident response procedures aligned with FERPA breach notification requirements.

22. Responsive Web Design

Mobile-optimized responsive design that works on all devices via browser. Mobile-optimized UI for common parent tasks. No native mobile apps for MVP - consider Progressive Web App (PWA) or native mobile apps in future based on user feedback.

Technical Stack

  • Frontend Framework — Nuxt.js with TypeScript for Vue-based frontend, server-side rendering for performance and SEO, component-based architecture with Vue 3
  • UI Framework — NuxtUI (based on Tailwind CSS) for consistent design system and component library
  • Database — MongoDB Atlas document database for flexible schema support, well-suited for form builder data and variable document structures, managed database service with automatic backups and scaling
  • State Management — Pinia for shared state that isn't tied to a single page's data
  • API Architecture — RESTful JSON API for all client-server communication using Nuxt's server routes (server/api/), standard HTTP methods and status codes, JWT-based authentication for API requests
  • Hosting & Infrastructure — Netlify for frontend hosting and deployment, serverless backend functions for API endpoints, CDN for static asset delivery, automatic SSL/TLS certificate management
  • Caching — Redis for session storage and application caching, CDN caching for static assets, database query caching for frequently accessed data
  • Email Provider — Mailgun integration for transactional emails with email templates, delivery tracking and bounce handling, rich text email support with attachments
  • SMS Provider — Twilio integration for SMS notifications with SMS templates and message scheduling, delivery status tracking, character limit enforcement and message splitting
