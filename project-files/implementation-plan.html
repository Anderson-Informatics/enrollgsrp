Wayne County PreK Enrollment System

Implementation Plan - MVP, Phase 2, and Future Features

Note: This implementation plan converts the high-level requirements from spec.html and features.html into actionable development phases with user stories, acceptance criteria, technical dependencies, and open questions.

MVP Feature Set

The MVP (Minimum Viable Product) focuses on core enrollment functionality required for the 2025-2026 school year. It enables end-to-end enrollment from parent application through student enrollment, with essential administrative capabilities.

MVP Scope:

  • Granular permission management system with preset permission sets and custom configurations for all admin user types (County Admin, District Admin, School Admin, Teacher)
  • Super Admin preset with all permissions enabled
  • Authentication (local + OAuth for parents, local + SAML/SSO for district staff)
  • County intake form with automatic eligibility determination and priority tier assignment
  • Form builder with comprehensive field types, conditional logic, validation rules, and multi-language support
  • District enrollment form built with form builder
  • Four-stage application lifecycle (Intake → Enrollment Paperwork → Enrolled → Site Visit) with parent portal status tracking
  • Document upload and verification workflow
  • PDF auto-fill for enrollment forms
  • Capacity and waitlist management
  • Teacher-parent communications (email, SMS, phone logging)
  • Site visit scheduling and ASQ assessment
  • Classroom assignment
  • Multi-language support (English, Spanish, Arabic) with RTL layout
  • Multi-child applications
  • Application transfers
  • Automated notifications
  • Reporting with enrollment statistics, waitlist reports, demographics, and contact activity metrics
  • CSV data export for SIS integration with future API integration support
  • Security and audit logging for FERPA compliance
  • Responsive web design

MVP Feature 1: Granular Permission Management System

User Stories

As a Super Admin, I want to have all permissions enabled so that I can perform any system-level task including managing all other users and their permissions.
As a County Admin, I want to create and manage District Admin accounts with preset permissions so that I can delegate district-level management responsibilities.
As a District Admin, I want to create School Admin and Teacher accounts with preset permissions within my district so that schools can manage their enrollment process.
As a District Admin, I want to create custom permission configurations for users so that I can define specialized roles (e.g., IEP Reviewer who can only verify IEPs but not modify permissions).
As a District Admin, I want to enable multi-school access for teachers and configure which schools they can access so that they can process applications from multiple schools when needed.
As a School Admin, I want to modify Teacher permissions at my school so that I can control what teachers can do without giving them full admin capabilities.
As a Parent, I want to self-register with email verification so that I can start the enrollment process with fixed permissions appropriate for my role.
Acceptance Criteria
  • Permission Presets: System includes Super Admin, County Admin, District Admin, School Admin, and Teacher presets with pre-configured permissions
  • Super Admin Preset: All permissions enabled, can manage all users and permissions across the system
  • County Admin Preset: System management permissions for all districts, can manage District Admins, School Admins, and Teachers
  • District Admin Preset: District management permissions for assigned district, can manage School Admins and Teachers within district
  • School Admin Preset: School management permissions for assigned school, can manage Teachers at school
  • Teacher Preset: Application processing, document verification, communication, and site visit permissions for assigned school(s)
  • Custom Permission Configuration: Admins can create custom permission sets by selecting individual permissions from logical groupings
  • Permission Groupings: Permissions organized into: System Management, District Management, School Management, Teacher, User Management, Data Access
  • Permission Scope: Permissions can be scoped to system-wide, district-wide, school-wide, or single school level
  • Permission Hierarchy: Higher-level users can modify permissions for lower-level users (Super Admin > County Admin > District Admin > School Admin > Teacher)
  • User Creation: When creating users, admins can select a preset or build custom permissions
  • Permission Modification: Higher-level admins can edit permissions of lower-level users through a permission modification interface
  • Permission Enforcement: System enforces permission checks on all data access and feature usage
  • Parent Permissions: Parent users have fixed permissions and cannot be modified
  • Audit Logging: All permission changes are logged with timestamp, user, and affected permissions
  • Access Control Indicators: UI shows available features based on current user's permissions
Technical Dependencies
  • Must be implemented before: Authentication System, any role-specific features
  • Requires: Database schema for users, roles, and permissions with many-to-many relationship
  • Requires: JWT authentication middleware with permission claims
  • Requires: Permission checking middleware/service for all API endpoints
  • Requires: UI permission checking service to show/hide features based on permissions
Open Questions
  • Should there be a permission template system for saving and reusing custom permission configurations?
  • How should we handle permission conflicts when a user has conflicting permissions from different groupings?
  • Should there be a permission request workflow where lower-level users can request additional permissions?
  • How should we handle bulk permission updates (e.g., updating all teachers in a district at once)?

MVP Feature 2: Authentication System

User Stories

As a Parent, I want to sign in with Google or Microsoft so that I don't have to remember another password.
As a Parent, I want to create a local username and password so that I can sign in without using external accounts.
As a District Staff Member, I want to sign in with my district SSO so that I don't have to remember separate credentials.
As a User, I want to reset my password via email so that I can regain access if I forget it.
Acceptance Criteria
  • Parents can authenticate via local username/password OR OAuth (Google/Microsoft)
  • District staff can authenticate via local username/password OR district SAML/SSO
  • County Admin can authenticate via local username/password with optional SSO configuration
  • Password reset flow works for local accounts via email verification
  • Passwords must meet minimum requirements (8+ chars, uppercase, lowercase, number, special character)
  • Session management with secure JWT tokens
  • Session timeout after inactivity (configurable)
  • District Admins can configure SAML/SSO settings for their district (app IDs, credentials)
  • OAuth and SAML accounts handle password resets through their identity providers
Technical Dependencies
  • Must be implemented before: Any authenticated features
  • Requires: User Role Management, OAuth provider accounts (Google/Microsoft), SAML/SSO library
  • Integrates with: Netlify Auth or custom auth service
Open Questions
  • Which SAML/SSO provider library should we use? (e.g., Auth0, Okta, Passport-SAML)
  • Should we support multiple OAuth providers beyond Google/Microsoft?
  • What should the session timeout duration be?
  • Should we implement multi-factor authentication for admin accounts?

  • Form supports multi-language (English, Spanish, Arabic) with RTL for Arabic

  • Form validation prevents submission with missing required fields
  • Parents can apply for multiple children with pre-filled family data
Technical Dependencies

  • Integrates with: Form Builder (for custom fields), Enrollment Priority System

  • How long should draft applications be retained before expiration?
  • Should there be a character limit on text fields?

MVP Feature 4: Automatic Eligibility Determination

User Stories

As a Parent, I want to know immediately if my child is eligible for enrollment so that I can understand the enrollment timeline.
As a School Admin, I want to see eligibility determinations for all applications so that I can prioritize reviews.
As the System, I want to automatically apply timing restrictions based on eligibility factors so that enrollment follows GSRP guidelines.
As the System, I want to automatically assign priority levels based on verified eligibility factors so that enrollment follows GSRP guidelines.
As a District Admin, I want to verify IEP documentation so that high priority status is correctly applied.
As a School Admin, I want to verify district custom priority responses so that sibling and other district-specific priorities are applied.
As a Parent, I want to understand my child's priority level so that I know their placement likelihood.
Acceptance Criteria
  • Age Eligibility: Child turns 4 on or before Sept 1 = eligible (no timing restriction)
  • Age Eligibility: Child turns 4 between Sept 2 - Dec 1 = conditionally eligible (hold until Sept 2 if space permits)
  • Priority Order (highest to lowest): Housing status → Verified IEP → Verified district custom priorities → FPL tiers
  • Housing Status: Homeless/foster care = highest priority regardless of FPL
  • IEP Status: Verified IEP = high priority regardless of FPL (requires admin verification)
  • District Custom Priorities: Priority between IEP and FPL tiers (requires admin verification)
  • FPL 0-100%: Priority Tier 1 (no timing restriction)
  • FPL 101-400%: Priority Tier 2 (no timing restriction)
  • FPL 401%+: Priority Tier 3 (hold until May 1)
  • Income Declined: Hold until August 1
  • System calculates FPL percentage from income and family size
  • District/School Admins can verify IEP documentation with status (Verified/Needs Review/Rejected)
  • District/School Admins can verify district custom priority responses
  • System applies enrollment holds based on priority and timing restrictions
  • Priority level displays on applicant records for admins
  • Priority level displays to parents in their dashboard
  • Eligibility results display to parents and route to school admins
  • Applications failing age eligibility are flagged for County Admin review
Technical Dependencies
  • Requires: County Intake Form, Document Upload & Verification
  • Must be implemented before: Application Lifecycle Stage 1 routing
  • Must be implemented before: Capacity & Waitlist Management
Open Questions
  • What FPL threshold values should be used? (federal poverty guidelines change annually)
  • How should the system handle leap years for DOB calculations?
  • Should there be a manual override option for County Admins on eligibility determinations?
  • What should happen if IEP verification is rejected? Does priority drop to FPL-based?
  • How should we handle district custom priority verification disputes?
  • Should priority be recalculated if verified information changes?

MVP Feature 5: Application Lifecycle Management

User Stories

As a Parent, I want to see my application's current stage and progress so that I know what to do next.
As a School Admin, I want to assign teachers to applications so that processing can begin.
As a Teacher, I want to transition applications through stages so that enrollment progresses properly.
As the System, I want to enforce stage transition rules so that the enrollment process follows the defined workflow.
As a Parent, I want to see document verification status so that I know which documents need attention.
As a Parent, I want to access communication history so that I can review all interactions.
As a Parent, I want to receive notifications for key events so that I stay informed.
Acceptance Criteria
  • Stage 1 (Intake): Parent submits intake form → School Admin assigns teacher → routes to school
  • Stage 2 (Enrollment Paperwork): Parent completes district form + uploads docs → Teacher verifies → all verified = transition to Stage 3
  • Stage 3 (Enrolled): Teacher marks enrolled → assigns classroom → student can attend
  • Stage 4 (Site Visit): Student attending + teacher calendar configured → parent schedules → teacher conducts ASQ
  • Application dashboard with stage indicator (Intake, Enrollment Paperwork, Enrolled, Site Visit)
  • Progress bar showing completion percentage through enrollment lifecycle
  • Available actions list based on current stage
  • "Waiting for" status indicating what parent is waiting for
  • Stage-specific details displayed:
    • Intake: form completion, document upload, teacher assignment notification
    • Enrollment Paperwork: form completion, document verification status, correction action items
    • Enrolled: classroom assignment, teacher contact, orientation details
    • Site Visit: teacher assignment, available time slots, scheduled visit, location confirmation
  • Document status dashboard showing Verified, Pending, Action Required for each document
  • Clear instructions for any needed corrections
  • Communication timeline showing all interactions chronologically
  • Notifications center for alerts and status updates
  • Multi-child application list view on parent dashboard
  • Switch between child applications from dashboard
  • Stage transitions trigger automated notifications to appropriate parties
  • System prevents invalid stage transitions (e.g., can't skip verification)
  • Teachers can view all applications at their assigned stage(s)
  • Admins can view applications across all stages
Technical Dependencies
  • Requires: All core features (forms, eligibility, documents, communications)
  • Requires: Multi-child Applications
  • Requires: State machine for application status transitions
Open Questions
  • Should applications be able to move backward in stages (e.g., from Enrolled back to Paperwork for corrections)?
  • What happens if a parent withdraws an application mid-process?
  • Should there be a "cancelled" or "withdrawn" status?
  • Should parents be able to download their application data?
  • Should parents be able to withdraw applications themselves?
  • How long should notification history be retained?

MVP Feature 6: Form Builder

User Stories

As a District Admin, I want to create custom enrollment forms so that I can collect district-specific information.
As a District Admin, I want to use drag-and-drop field placement so that form creation is intuitive.
As a District Admin, I want to set validation rules on fields so that submitted data is accurate.
As a Parent, I want to complete forms in multiple languages so that I can use my preferred language.
Acceptance Criteria
  • Supported field types: Text, Textarea, Number, Date, Email, Phone, Address block, Single Select, Radio, Checkbox, Multi-select, File Upload

  • Admins can configure field properties: label, placeholder, required, help text, validation rules
  • Admins can create conditional show/hide logic based on previous field values
  • Forms support multi-language labels (English, Spanish, Arabic) with RTL layout for Arabic
  • Admins can preview forms before publishing
  • Forms can be versioned (new versions preserve existing submissions)
  • Field validation includes: required, min/max length, pattern matching, date ranges
  • File upload fields accept PDF, JPG, PNG with size limits

Technical Dependencies
  • Requires: Authentication, User Role Management
  • Must be implemented before: District Enrollment Form
  • Requires: Multi-language Support infrastructure
Open Questions
  • What is the maximum number of fields per form?
  • Should there be a limit on conditional logic complexity?
  • How should we handle form deletion when submissions exist?

MVP Feature 8: Document Upload & Verification

User Stories

As a Parent, I want to upload required documents so that my child's enrollment can be verified.
As a Parent, I want to upload photos from my phone and have them converted to PDF so that I can submit documents easily.
As a Teacher, I want to review uploaded documents and mark them as verified or needing correction so that enrollment can proceed.
As a Teacher, I want to request specific document corrections with explanations so that parents know exactly what to fix.
Acceptance Criteria
  • Parents can upload documents during enrollment paperwork stage
  • Accepted file types: PDF, JPG, PNG with size limits (e.g., 10MB max)
  • Built-in photo-to-PDF converter for mobile devices when uploading images
  • System validates file types and displays upload confirmation
  • Teachers review each document and mark as: Verified, Needs Correction, or Missing
  • Teachers can request specific corrections with text explanations
  • Parents receive notifications when corrections are needed
  • Parents can upload corrected documents
  • Real-time status display for parents (Verified, Pending, Action Required)
  • Audit trail of all document verifications with timestamps and reviewer
  • Common document types: proof of residency, income verification, immunization records, custody documents
  • Teachers can download documents for review
Technical Dependencies
  • Requires: District Enrollment Form, Application Lifecycle Management
  • Must be implemented before: PDF Auto-Fill, Stage 2 completion
  • Requires: Netlify Blob for document storage, image-to-PDF conversion library
Document Storage Architecture (Netlify Blob)

Storage Solution: Netlify Blob for object storage with application-level access control

Architecture Pattern:

  • Storage: Documents stored in Netlify Blob with CDN delivery
  • Metadata: Document ownership (applicationId, studentId, schoolId, districtId) stored in MongoDB
  • Access Control: Server routes check user permissions before serving files (never expose direct Blob URLs)
  • Permission Enforcement: Hierarchical access based on user roles (Parents → own documents, Teachers → assigned school(s), School Admins → their school, District Admins → district schools, County Admins → system-wide)
  • Security: Signed URLs with expiration for temporary access, server-side proxy for all file requests

Implementation:

  • Create server routes like /api/documents/[documentId] that check permissions before fetching from Blob
  • Generate short-lived signed URLs after permission verification
  • Reference MongoDB metadata to validate user has access to requested document
  • Stream file content from Blob storage through authenticated server route
Open Questions
  • What file size limit should we enforce?
  • Should documents be virus-scanned on upload?
  • How long should documents be retained after enrollment is complete?
  • Should there be a document expiration or renewal requirement?

MVP Feature 10: Capacity & Waitlist Management

User Stories

As a District Admin, I want to configure capacity for each school so that enrollment is managed appropriately.
As the System, I want to automatically route applications to first-choice schools with available capacity so that placement is efficient.
As the System, I want to automatically maintain waitlists for full schools so that spots are filled by priority when they open.
As a School Admin, I want to view capacity status and waitlist positions so that I can manage enrollment effectively.
Acceptance Criteria
  • District Admins configure capacity for each school in their district
  • System automatically routes applications to first-choice school with available capacity
  • If first choice is full, routes to next available choice in parent's ranked list
  • Applications for full schools placed on automated waitlists
  • Waitlists ordered by priority (housing > IEP > district custom > FPL tiers)
  • When spots open, system fills from waitlist based on priority order
  • System maintains 10% allocation hold for 0-400% FPL students until September 1
  • Waitlist must be maintained and reportable for Spring Early Childhood Collection
  • Subrecipients can refer waitlisted families to (313) 410-4588 or FindPREK.org
  • Capacity status dashboard showing enrollment vs. available spots
  • Automated waitlist movement notifications to parents when spots open
  • Students absent 2 weeks without notice are removed and placed at top of waitlist for their site
Technical Dependencies
  • Requires: Enrollment Priority System, Application Lifecycle Management
  • Must be implemented before: Application routing logic
Open Questions
  • Should capacity be configurable per classroom or per school?
  • How should we handle over-enrollment if capacity is exceeded?
  • Should waitlist position be visible to parents?
  • What happens if a parent declines a waitlist offer?
